Estudo de Caso: KDD Cup 2009¶

A Administrao de Relacionamento com o Consumidor um elemento-chave das estratgias modernas de marketing. A maneira mais prtica de adquirir conhecimento sobre consumidores produzindo pontuaes. Uma pontuao a sada de um modelo e serve como avaliao de todas as instncias de uma varivel-alvo que se pretenda explicar. Ferramentas que produzem pontuaes permitem projetar informaes quantificveis em uma dada populao. A pontuao calculada usando variveis de entrada que descrevem as instncias. As pontuaes so ento usadas pelo sistema de informao, por exemplo, para personalizar o relacionamento com o consumidor.

Em 2009, a conferncia Knowledge Discovery and Data Mining (KDD) lanou um desafio envolvendo aprendizagem de mquina. Ela disponibilizou dados de consumidores da empresa de telecomunicaes francesa Orange, que deveriam ser utilizados para a construo de um modelo capaz de prever trs comportamentos do consumidor. A planilha fornecida tem 50000 instncias e 230 variveis, sendo as primeiras 190 variveis numricas e as demais 40 categricas. As variveis de sada do modelo eram o cancelamento da conta ('churn'), a tendncia de usar novos produtos e servios ('appetency') e a propenso para comprar upgrades ou adicionais com maior margem de lucro quando apresentados (upselling), fornecidos de forma binria (-1 para no, 1 para sim).

O trabalho apresentado aqui envolve a criao de um modelo para predio de churn, appetency e upselling com base nos dados fornecidos dos clientes da Orange. O critrio para avaliao do modelo ser a mtrica AUC, que dimensiona o trade-off entre taxa de falsos positivos e taxa de verdadeiros positivos: quanto mais prximo de 1, menor o trade-off e melhor o modelo.

Detalhes sobre a competio: https://www.kdd.org/kdd-cup/view/kdd-cup-2009

def etc_importances(features, outcome):
  # Parecido com um random_forest
  ## As vezes esse classificador sai melhor que o random_forest
  ### n_estimator : qtd de rvores para comparar
  ### criterion: critrio de clasificao
  ### verbose: 0 para no mostrar nada
  model = ExtraTreesClassifier(
      n_estimators = 200, 
      criterion = 'entropy', 
      verbose = 0)
  
  # O ravel  para retornar no formato de vetor/lista
  model.fit(features, np.array(outcome).ravel())
  
  importances = model.feature_importances_ 
    # Rankeamento de importancia em ordem desordenada
    # Cada varivei  uma porcentagem, a soma de todas elas  1
  variables = np.array(features.columns)
  indices = np.argsort(importances)[::-1] 
    # Ordena em ordem crescente e inveter (Do maior para o menor : decrescente)
    # argsort: faz o sort, mas, em vez de retornar valor, retorna a ordem dos index ordenados
  importances = importances[indices] # Coloca as importanves em ordem
  variables = variables[indices] #Retorna as variaveis na ordem de importancia
  
  return variables, importances

Conclusões¶

Nesse exerccio, foi possvel modelar a possibilidade do cancelamento da conta com uma rea sob a curva (AUC) de 0.6868

bom lembrar que AUC = 0.5 indica a linha de base, no caso de um modelo que faz predies aleatrias, e AUC = 1.0 indica um modelo ideal, que faz predies perfeitas

Pode parecer que o AUC alcanado no muito alto, mas tambm temos que considerar que o score mais alto no desafio pblico foi de 0.76

Isso indica que, muito possivelmente, os dados fornecidos no so suficientes para fazer a modelagem perfeita do cancelamento da conta, e isso sempre algo que devemos levar em considerao quando estamos tentando desenvolver modelos para problemas que no temos qualquer conhecimento prvio

Fazendo uma analogia simples, seria como tentar predizer o peso de uma pessoa usando uma base de dados com apenas peso e altura de uma populao: ns at conseguiramos uma resposta razovel, j que as duas variveis so correlacionadas, mas ela no seria perfeita simplesmente porque mais variveis tm impacto no peso alm da altura. possvel, portanto, que o maior score indique o melhor modelo que esses dados so capazes de fornecer

Tambm devemos nos lembrar de que esses desafios geralmente so lanados sem que exista uma resposta prvia, eles servem justamente para que a prpria comunidade tente encontrar solues e comparar seus resultados entre si, e no com um "padro ouro" ou a "resposta certa"

Esses desafios so uma tima oportunidade para se familiarizar com o ambiente real da carreira de um cientista de dados, e para conhecer e implementar na prtica mtodos e abordagens novas.

Melhorias¶

Testar outros modelos sempre uma opo que se deve considerar. Nesse exerccio, ns paramos nos classificadores do tipo gradient boosting, mas tambm poderamos ter considerado modelos baseados em rvores de deciso mais novos, como o XGBoost, e at em redes neurais

Os dados esto mascarados, o que dificulta qualquer feature engineering, j que para isso geralmente ns tentamos transformar as variveis em funo do conhecimento prvio que temos sobre elas. Por exemplo, se tivssemos datas, poderamos criar uma nova varivel refletindo as estaes do ano, se julgssemos relevante, mas nesse exerccio ficamos restritos a tratar os dados com mtricas estritamente numricas, como o nmero de categorias que uma varivel categrica possui, por no sabermos a que a varivel se refere

Mesmo com essas restries, a seleo das melhores variveis pode ser feita apenas considerando a natureza dos dados, e no da prpria varivel, e possvel testar outros mtodos de seleo alm daquele que ns usamos (seleo baseada em rvore de deciso)

Tambm possvel melhorar o processo de fine-tuning da seguinte maneira: primeiro ns treinamos um hiperparmetro num intervalo maior, por exemplo, [1, 2, 3, 4, 5], e depois que ns descobrimos o melhor parmetro nessa faixa, por exemplo [2], podemos treinar de novo nas imediaes desse valor, por exemplo [1.5, 2, 2.5], e assim por diante

Ao final do treinamento apresentado no exerccio (MODELO 8), ns observamos que a mtrica continuava em tendncia de aumentar, ento talvez seja interessante continuar testando outros valores para learning rate/nmero de pocas

Tambm til fazer um grid search com todos os parmetros ao mesmo tempo, j que no necessariamente cada parmetro se comporta da forma independente que ns consideramos, mas para isso bom usar poucos valores para cada parmetro (por isso bom fazer uma seleo prvia da forma que ns fizemos), ou teramos um nmero de modelos excessivamente grande pra testar, o que demanda maior capacidade de processamento
